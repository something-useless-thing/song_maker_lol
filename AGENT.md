# AGENT Rule

- 기본적으로 최소한의 변경으로 목적을 달성하게 해줘. 요청하지 않은 리팩토링/파일 구조 변경은 하지 말고, 하고 싶으면 먼저 물어봐줘.
- 확장성을 고려하되 지금 있는 디렉토리 구조와 네이밍 규칙을 최대한 지켜줘. 구조를 바꿔야 할 것 같으면 이유를 설명하고 허락을 먼저 받아줘.
- 코드에 주석을 상세하게 달아줘 — 특히 오디오 타이밍/스케줄링처럼 직관적이지 않은 부분은 "왜 이렇게 했는지"까지 설명해줘.
- 이 프로젝트는 보안/개인정보를 다루지 않는 프론트엔드 창작 툴이지만, API 키나 외부 서비스 키를 쓰게 되면 절대 프론트 코드에 하드코딩하지 말고 환경변수로 관리해줘.

## 프로젝트 개요

- 이름: Music Maker (가칭이었던 "뮤직랩"에서 확정)
- 무엇: 브라우저에서 그리드를 클릭해서 MIDI 패턴을 만들고, 만든 패턴을 여러 악기로 재생해볼 수 있는 미니 DAW. BandLab/FL Studio를 참고한 화이트모드, 헤더+트랜스포트바 구조.
- 기획 문서: `docs/PRD.md`
- 디자인 시스템: `docs/DESIGN.md` (색상/타이포/컴포넌트 스펙, 헤더·트랜스포트바 레이아웃 규칙 포함 — 화면 만들 때 반드시 참고)

## 기술 스택

- Vite + React + TypeScript 기반으로 작업해줘.
- 오디오 재생/스케줄링은 Tone.js를 사용해줘. setTimeout 기반 타이밍은 쓰지 말고 반드시 Tone.Transport 스케줄러를 사용해줘 — 그렇지 않으면 재생 타이밍이 밀림.
- 악기 소리는 기본 신스 대신 프로젝트에 준비된 wav 샘플을 `Tone.Sampler`로 로드해서 사용해줘. 샘플 경로는 `public/samples/` 아래에 악기별 폴더로 정리해줘.
- 스타일링은 CSS Modules 또는 plain CSS로, `docs/DESIGN.md`에 정의된 컬러/타이포/스페이싱 토큰을 CSS 변수로 옮겨서 재사용해줘 (예: `--color-primary`, `--radius-md` 등). Tailwind를 쓰고 싶으면 제안해주고 확인받은 후 도입해줘.
- 상태 관리는 별도 라이브러리 없이 React 기본 상태(useState/useReducer/Context)로 충분함. 프로젝트가 커지기 전까지 Zustand 등 추가 라이브러리 도입은 제안만 하고 바로 넣지 말아줘.

## 코딩 스타일

- TypeScript any 타입 쓰지 말아줘 — eslint에서 에러 나지 않게 타입을 제대로 정의해줘.
- 함수/컴포넌트의 반환 타입을 명시적으로 작성해줘.
- 오디오 관련 로직(그리드 좌표 ↔ 음높이/박자 매핑, Tone.js 스케줄링)은 별도 훅(`useSequencer` 같은)으로 분리해서 UI 컴포넌트와 분리해줘.
- 새 컴포넌트를 만들 때는 `docs/DESIGN.md`의 컴포넌트 섹션에 있는 이름(`grid-cell`, `transport-play-button` 등)과 최대한 맞춰서 컴포넌트/클래스명을 지어줘 — 기획 문서와 코드가 서로 다른 용어를 쓰지 않게.

## 지금 우선순위 (MVP)

`docs/PRD.md`의 REQ-01~03 (스텝 시퀀서 그리드 + 악기 선택 + 재생 컨트롤)을 가장 먼저 완성해줘. REQ-04(플레이리스트)와 REQ-05(저장/공유)는 시간이 남으면 진행하는 확장 기능임 — 먼저 완성도 있게 돌아가는 MVP부터 만들어줘.
